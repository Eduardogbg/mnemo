> Research date: 2026-03-12
> Context: Mnemo negotiation system — 2-agent DAG conversation with scope rewind/fork
> Infrastructure: Phala GPU-TEE (H100/H200/B200), SGLang/vLLM, self-hosted or Redpill API

## A. KV Cache Fundamentals (With Real Numbers)

### The Formula

For standard GQA (Grouped Query Attention) models:

```
kv_bytes_per_token = 2 * num_layers * num_kv_heads * head_dim * bytes_per_element
```

Where:
- `2` = one K tensor + one V tensor
- `num_layers` = transformer depth
- `num_kv_heads` = number of key-value heads (fewer than query heads in GQA)
- `head_dim` = dimension per attention head
- `bytes_per_element` = 2 for FP16/BF16, 1 for FP8/INT8

For MLA (Multi-head Latent Attention, used by DeepSeek V3):

```
kv_bytes_per_token = num_layers * (kv_lora_rank + qk_rope_head_dim) * bytes_per_element
```

MLA compresses KV into a low-rank latent representation rather than storing per-head K/V directly.

### Model Architecture Specs

| Model | Architecture | Layers | KV Heads | Head Dim | Total Params | Active Params |
|-------|-------------|--------|----------|----------|-------------|---------------|
| **Llama 3.1 70B** | Dense, GQA | 80 | 8 | 128 | 70B | 70B |
| **Qwen3 30B-A3B** | MoE, GQA | 48 | 4 | 128 | 30.5B | 3.3B |
| **DeepSeek V3** | MoE, MLA | 61 | N/A (MLA) | N/A (MLA) | 671B | 37B |
| **GPT-OSS 120B** | MoE, GQA | 36 | 8 | 64 | 117B | 5.1B |

Notes on DeepSeek V3 MLA parameters: `kv_lora_rank=512`, `qk_rope_head_dim=64`, `num_attention_heads=128`.

| **GPT-OSS 120B** | 144 MB | 288 MB | 576 MB | 1.15 GB |

### Does MoE Affect KV Cache Size?

**No.** KV cache is generated by the attention layers, not the feed-forward/expert layers. In MoE architectures:

- All tokens pass through all attention layers (attention is NOT sparse/routed)
- MoE routing only applies to the feed-forward (FFN) blocks
- Therefore KV cache size is determined solely by attention architecture (num_layers, num_kv_heads, head_dim)
- A MoE model with the same attention config as a dense model will have identical KV cache per token

The exception is DeepSeek V3, which uses MLA (Multi-head Latent Attention) — this is an attention-level optimization orthogonal to MoE. MLA compresses KV cache regardless of whether the FFN is dense or MoE.

**Key insight for Mnemo:** MoE models give us the best of both worlds for KV caching — they have small KV caches (because they tend to use fewer layers/heads than equivalent dense models) AND small active compute per token. GPT-OSS 120B and Qwen3 30B-A3B are standouts: frontier-class reasoning with KV caches 3-4x smaller than Llama 70B.

## B. The DAG Advantage: Why Our Pattern Is Cheaper

### The Core Thesis

Normal API inference charges per-token because each request recomputes the full context. With prefix caching (APC in vLLM, RadixAttention in SGLang), repeated prefixes are computed once and reused. Mnemo's DAG structure naturally produces high prefix overlap because:

1. **Scope rewind** snaps context back to a checkpoint — the checkpoint prefix is identical to what was previously computed
2. **Agent fork** sends two requests with the same prefix and different suffixes
3. **Normal turn progression** extends the previous context by one message

### Concrete Scenario

**Parameters:**
- 30 turns total
- Average 150 tokens per message (output)
- Average 150 tokens per message when read as input (with role tags etc.)
- System prompt: 500 tokens
- 3 scope open/close cycles (scope opens at turns 8, 15, 22; closes at turns 12, 19, 27)
- Each scope close rewinds to ~60% of current context
- 1 agent fork at turn 20 (2 parallel requests sharing prefix)

### Token Accounting: WITHOUT Prefix Caching

Every API call sends the full context and the provider recomputes all of it.

```
Turn 1:  500 + 150 = 650 tokens processed
Turn 2:  500 + 300 = 800
Turn 3:  500 + 450 = 950
...
Turn N:  500 + (N * 150) = 500 + 150N
```

For 30 turns (simplified, ignoring scope deletions temporarily):

Sum of input tokens = sum(500 + 150*i for i in 1..30)
                    = 30*500 + 150 * (30*31/2)
                    = 15,000 + 150 * 465
                    = 15,000 + 69,750
                    = 84,750 prompt tokens processed

With scope rewinds (3 cycles, each removing ~40% of accumulated context at the point of rewind):
- Scope closes reduce the running context, but each subsequent turn still rebuilds from the surviving prefix
- Net effect: scope rewinds reduce the average context length by ~15-20% across the session
- Estimated total with rewinds: ~72,000 prompt tokens processed

Plus the fork (2 parallel requests of ~3,000 tokens each): +6,000
Plus 30 output generations at ~150 tokens each: 4,500 output tokens

**Total WITHOUT caching: ~78,000 prompt tokens processed + 4,500 output tokens**

### Token Accounting: WITH Prefix Caching

With prefix caching, each turn only computes the new suffix (the latest message). The prefix KV cache is reused from the previous turn.

**Normal turns (append-only):**
Each turn computes only the new message: ~150 tokens of fresh compute. The prefix (all prior messages) is a cache hit.

**Scope rewind turns:**
After scope close, the context snaps back to checkpoint. If the checkpoint prefix is still cached (likely — it was used moments ago), only the new post-rewind message is computed: ~150 tokens fresh. Without caching, this would recompute the entire checkpoint prefix (~1,500-3,000 tokens).

**Fork:**
Two requests sharing a ~3,000-token prefix. With caching: prefix computed once (3,000 tokens), each fork adds ~150 tokens suffix. Without caching: 2 * 3,000 = 6,000 tokens.

```
Fresh tokens computed WITH caching:
- 30 turns * 150 tokens (new message each turn)     = 4,500
- 1 initial context build (turn 1, full system+msg)  = 650
- 3 scope rewinds (cache hit on prefix, fresh suffix) = 3 * 150 = 450
- 1 fork (prefix cached, 2 suffixes)                  = 300
                                                     --------
Total fresh compute:                                   ~5,900 tokens

Cached (prefix hit) tokens:
= 78,000 - 5,900 = ~72,100 tokens (cache hits)
```

### Cost Comparison

**Using Redpill DeepSeek V3 pricing ($0.28/M input, $1.14/M output):**

| Scenario | Prompt Tokens Processed | Fresh Compute | Cost (Input) | Cost (Output) | Total |
|----------|------------------------|---------------|-------------|---------------|-------|
| No caching | 78,000 | 78,000 | $0.022 | $0.005 | **$0.027** |
| With prefix caching | 78,000 nominal, 5,900 fresh | 5,900 | $0.002 | $0.005 | **$0.007** |

**Cost reduction: ~75% on input tokens, ~74% overall.**

But wait — at these absolute numbers, the difference is $0.020. Two cents. Per negotiation.

**At higher volume (1,000 negotiations/day):**

| Scenario | Daily Input Cost | Daily Output Cost | Daily Total |
|----------|-----------------|-------------------|-------------|
| No caching | $21.84 | $5.13 | $26.97 |
| With caching | $1.65 | $5.13 | $6.78 |
| **Saving** | | | **$20.19/day** |

**The savings from prefix caching are real but only materially significant at 500+ negotiations per day.** At hackathon scale (10-50 negotiations/day), the difference is under $1/day.

### Where Caching DOES Matter: Latency

The bigger win is latency, not cost. With prefix caching:
- Time-to-first-token (TTFT) drops by 2-5x on cached prefixes
- A 3,000-token prefix that takes ~2-4 seconds to prefill becomes near-instant (~100ms)
- Scope rewinds feel instantaneous instead of taking several seconds
- Forks return faster because the shared prefix isn't recomputed

For a real-time negotiation demo, this latency improvement matters more than the cost savings.

## C. Interleaving Economics

### The Insight

A negotiation is fundamentally turn-based: while Agent A generates a response, Agent B is idle (and vice versa). If we control the inference server, we can interleave requests from multiple negotiations on the same GPU to maximize utilization.

### How Many Concurrent Negotiations Fit in GPU Memory?

**Model: GPT-OSS 120B (MoE, FP16)**

| Component | Memory |
|-----------|--------|
| Model weights (FP16, all experts loaded) | ~30 GB (estimated from 117B params but MoE sharding — need all experts in VRAM) |
| CUDA overhead, activations, buffers | ~5 GB |
| **Available for KV cache** | **~45 GB** |

KV cache per negotiation session at average context length (~3,000 tokens):
- GPT-OSS 120B: 72 KB/token * 3,000 = 216 MB per session

**Concurrent sessions in 45 GB: ~208 sessions.**

That's far more than we'd ever need. Even at 16K tokens per session (deep negotiations): 1.15 GB per session, ~39 concurrent sessions.

**Model: Llama 3.1 70B (AWQ INT4)**

| Component | Memory |
|-----------|--------|
| Model weights (INT4) | ~35 GB |
| CUDA overhead | ~5 GB |
| **Available for KV cache** | **~40 GB** |

KV cache per session (3,000 tokens): 320 KB * 3,000 = 960 MB per session.

**Concurrent sessions in 40 GB: ~41 sessions.**

At 8K tokens per session: 2.56 GB, ~15 concurrent sessions.

**Model: Qwen3 30B-A3B (FP16)**

| Component | Memory |
|-----------|--------|
| Model weights (FP16) | ~8 GB (estimated — 30.5B params but only need routing + all experts loaded) |
| CUDA overhead | ~3 GB |
| **Available for KV cache** | **~69 GB** |

KV cache per session (3,000 tokens): 96 KB * 3,000 = 288 MB.

**Concurrent sessions in 69 GB: ~239 sessions.**

### Utilization Improvement from Interleaving

**Sequential (one negotiation at a time):**
- A typical negotiation turn takes ~2-5 seconds for inference
- Between turns: orchestration overhead, scope management, message routing = ~0.5-1 seconds
- GPU is idle during orchestration overhead
- Estimated GPU utilization: ~60-70%

**Interleaved (multiple negotiations):**
- While Negotiation A's orchestration runs, Negotiation B's inference runs
- With 3+ concurrent negotiations, there's almost always a request queued
- vLLM/SGLang continuous batching handles this automatically
- Estimated GPU utilization: ~85-95%

**Utilization improvement: ~30-50% more effective throughput from the same GPU.**

### Batching Across Negotiations with Shared System Prompts

If multiple negotiations use the same system prompt (likely — Mnemo has a standard agent personality), the system prompt KV cache is computed once and shared across all sessions. With SGLang's LFU eviction, this system prompt stays pinned.

For 10 concurrent negotiations sharing a 500-token system prompt:
- Without sharing: 10 * 500 * 72 KB = 360 MB of redundant KV cache
- With sharing: 500 * 72 KB = 36 MB (one copy, referenced 10x)
- Saving: 324 MB (~0.3 GB) — small but meaningful when fitting more sessions

## D. Checkpoint Strategy

### Natural Checkpoint Points

Mnemo's scope boundaries are natural checkpoints. At each scope open, the current context prefix becomes a potential rewind target. Our strategy:

1. **Pin checkpoint KV caches** at scope boundaries (the prefix up to that point)
2. **On scope close**: drop the scope's KV blocks (tokens inside the scope), keep the checkpoint prefix
3. **On fork**: copy-on-write from the checkpoint KV cache (both branches share the prefix)

### Memory Cost of Checkpoint Strategy

**Scenario: 3 checkpoints at positions [500, 1200, 2000] tokens using GPT-OSS 120B (72 KB/token):**

The checkpoints are nested — checkpoint 2 includes all of checkpoint 1. So the actual stored data is:

| Checkpoint | Token Position | Incremental KV Data | Cumulative KV Data |
|-----------|---------------|--------------------|--------------------|
| CP1 | 500 | 500 * 72 KB = 36 MB | 36 MB |
| CP2 | 1200 | 700 * 72 KB = 50.4 MB | 86.4 MB |
| CP3 | 2000 | 800 * 72 KB = 57.6 MB | 144 MB |

**Total checkpoint memory: 144 MB** (only CP3 is the full dataset; CP1 and CP2 are subsets already contained within CP3's KV cache).

Actually, with a radix tree (SGLang) or block-based cache (vLLM), these are automatically shared — the blocks for tokens 0-500 are the same physical memory referenced by all three checkpoints. So the true memory footprint is just the KV cache for the longest checkpoint: **144 MB**.

**With Llama 70B (320 KB/token):**
- CP3 at 2,000 tokens: 640 MB
- Still very manageable on an H100 with 40-45 GB available for KV cache

### Copy-on-Write for Forks

When forking from a checkpoint, both branches start with the same KV cache blocks. New tokens generated in each branch allocate fresh blocks. The checkpoint blocks are never modified (attention KV cache is append-only by nature).

In practice, both vLLM (PagedAttention with refcounting) and SGLang (radix tree) implement this naturally — no custom copy-on-write logic needed. The shared prefix blocks have refcount > 1 and are never evicted while any request references them.

## E. MoE Model Economics

### The MoE Advantage for Inference

| Property | Dense (Llama 70B) | MoE (GPT-OSS 120B) | MoE (Qwen3 30B-A3B) |
|----------|-------------------|---------------------|----------------------|
| Total params | 70B | 117B | 30.5B |
| Active params/token | 70B | 5.1B | 3.3B |
| Model weights in VRAM (FP16) | ~140 GB | ~30 GB* | ~8 GB* |
| Model weights (INT4) | ~35 GB | N/A (already fits) | N/A (already fits) |
| KV cache/token | 320 KB | 72 KB | 96 KB |
| Compute per token (relative) | 1.0x | ~0.07x | ~0.05x |
| VRAM for KV cache (on H100) | ~40 GB | ~45 GB | ~69 GB |

*MoE weight estimates: all experts must be loaded even though only a subset fire per token. The VRAM for model weights is proportional to total params, but MoE models pack more total params into fewer active params. The exact VRAM depends on the expert structure.

**Correction on MoE VRAM:** GPT-OSS 120B has 117B total parameters. At FP16, that's ~234 GB — does NOT fit on a single H100. However, with FP8 or INT4 quantization:
- FP8: ~117 GB — still too large for 1x H100 (80 GB)
- INT4: ~58 GB — fits, but tight (leaves ~22 GB for KV cache)

This is important. Let me recalculate based on the earlier research document which stated GPT-OSS 120B at FP16 requires ~30 GB. That estimate likely comes from the *active* parameters (5.1B * 2 bytes = ~10 GB active, but all experts still need to be in VRAM). Looking more carefully:

**GPT-OSS 120B reality check:**
- 36 layers, 128 experts per layer, top-4 routing
- Total experts: 36 * 128 = 4,608 experts
- Each expert is small (total 117B / structured as shared + expert params)
- The research doc's "~30 GB" estimate for FP16 seems too low for 117B params
- More realistic: 117B * 2 bytes = 234 GB at FP16, need multi-GPU or heavy quantization
- At INT4: ~58 GB, fits on H100 with ~22 GB for KV cache
- **However**: Redpill runs GPT-OSS on their own multi-GPU clusters, so this isn't our problem when using the API

**Revised single-H100 fit check:**

| Model | Quantization | VRAM for Weights | VRAM for KV Cache | Fits H100 80GB? |
|-------|-------------|-----------------|-------------------|-----------------|
| GPT-OSS 120B | FP16 | ~234 GB | N/A | **NO** |
| GPT-OSS 120B | INT4 | ~58 GB | ~22 GB | YES (tight) |
| Qwen3 30B-A3B | FP16 | ~61 GB* | ~19 GB | YES (tight) |
| Qwen3 30B-A3B | INT8 | ~30 GB | ~50 GB | **YES (good)** |
| Llama 3.1 70B | INT4 (AWQ) | ~35 GB | ~45 GB | **YES (good)** |
| Llama 3.1 70B | FP8 | ~70 GB | ~10 GB | YES (very tight) |

*Qwen3 30B-A3B: 30.5B total params * 2 bytes = 61 GB at FP16. Fits on H100 but leaves limited KV cache room. At INT8 (~30 GB) it's much better.

### Are MoE Savings Additive with Prefix Caching?

**Yes, they are independent and additive.** Here's why:

- **MoE saves compute** per token: only active experts fire in the FFN layers. This reduces FLOPs per token by (active/total) ratio for the FFN portion. The attention layers run at full cost regardless.
- **Prefix caching saves compute** on the attention layers: cached prefix tokens skip the attention computation entirely (the KV is already computed).

For a cached prefix token in a MoE model:
- Attention: **0 compute** (cache hit — KV already exists)
- FFN: **0 compute** (the entire layer is skipped when KV is cached — FFN runs during prefill, not during cache lookup)

Wait — this needs clarification. During prefill, both attention AND FFN run for each token. Prefix caching skips the entire forward pass for cached tokens (both attention and FFN). So:

- **Without prefix caching, MoE model**: each token's forward pass costs attention (full) + FFN (sparse). MoE saves ~60-95% on the FFN portion.
- **With prefix caching, MoE model**: cached prefix tokens cost 0 (entire forward pass skipped). New suffix tokens cost attention (full) + FFN (sparse).

**The savings compound:**
- Prefix caching eliminates cost for ~75% of tokens (in our scenario)
- MoE reduces cost for the remaining ~25% of tokens by ~60-95% on FFN
- Net effect: roughly 75% + (25% * 60-95% FFN saving * FFN fraction of total compute)

For a typical transformer, FFN is roughly 2/3 of total FLOPs, attention is 1/3:
- MoE saving on new tokens: ~60-95% * 2/3 = 40-63% of new-token compute
- Combined: 75% saved by caching + (25% * 40-63%) = 75% + 10-16% = **85-91% total compute reduction**

vs. just prefix caching on a dense model: 75% saved, 25% at full cost = 75% reduction.

**MoE + prefix caching is roughly 10-16 percentage points more efficient than prefix caching alone.**

## F. Apple Silicon Footnote

Apple M-series SoCs have unified memory architecture (UMA) — CPU, GPU, and Neural Engine share a single pool of high-bandwidth memory (up to 128 GB on M4 Max, 192 GB on M4 Ultra).

### Implications for DAG-Based Agents

- **No PCIe bottleneck**: KV cache lives in the same memory pool as model weights. No data transfer between CPU RAM and GPU VRAM — both access the same physical memory.
- **Larger effective "VRAM"**: An M4 Max with 128 GB can hold a 70B model (INT4, ~35 GB) plus massive KV caches (~90 GB available). This is more KV cache room than an H100.
- **Copy-on-write forking is trivially cheap**: macOS's virtual memory system supports COW natively. Forking a process that holds KV cache in memory-mapped files creates a fork with shared KV state at zero copy cost.
- **mlx framework**: Apple's mlx can run LLM inference efficiently on M-series with prefix caching support.

### Cost Comparison for Personal Use

| Platform | Hardware Cost | KV Cache Capacity | Ongoing Cost |
|----------|-------------|-------------------|-------------|
| M4 Max (128 GB) | ~$4,000 (one-time) | ~90 GB | Electricity only (~$0.05/hr) |
| H100 80 GB (cloud) | $0 | ~45 GB | $3.08/hr |
| API (Redpill) | $0 | Provider-managed | ~$0.03/negotiation |

For personal/local agent use, Apple Silicon is potentially the cheapest option for DAG-based inference after ~130 hours of use (breakeven vs. H100 rental). However, **Apple Silicon cannot provide TEE guarantees** — there is no equivalent to Intel TDX + NVIDIA CC on Apple hardware.

**Verdict**: Interesting for local development and personal agent use. Not viable for Mnemo's trust model which requires GPU-TEE. Filed as a future direction for "trusted personal agent" scenarios where the user trusts their own hardware.

## H. Bottom Line Recommendations

### At What Scale Do "Caching Shenanigans" Actually Matter?

**Economically (cost):**
- At <100 negotiations/day: prefix caching saves <$2/day. **Does not matter.**
- At 1,000 negotiations/day: saves ~$20/day. **Noticeable but not decisive.**
- At 10,000+ negotiations/day: saves ~$200/day. **Meaningful.**

**Latency-wise:**
- At any scale: prefix caching reduces TTFT by 2-5x on rewinds/forks. **Always matters for UX.**
- Scope rewinds go from 2-4 second pauses to <0.5 seconds. **Perceptible improvement for demo.**

**Bottom line on cost: prefix caching is not about saving money at our scale. It's about latency.**

### Minimum Viable Strategy for the Hackathon

1. **Use Redpill API** (GPT-OSS 120B or DeepSeek V3). Cost: under $1 for the entire hackathon.
2. **Structure prompts correctly**: stable prefix first (system prompt, then chronological messages), variable content last. This maximizes cache hit probability *if* Redpill has APC enabled.
3. **Run the empirical TTFT test** (from research-redpill-kv-caching.md) to detect if caching is active.
4. **Don't self-host unless** the TTFT test shows no caching AND latency is unacceptable for the demo.
5. **Have SGLang Docker Compose ready** as a fallback (from research-self-hosted-inference.md).

**Estimated total inference cost for hackathon: $1-5.**

### Production Strategy If Mnemo Scales

**Phase 1 (0-100 negotiations/day):** Redpill API. Cost: <$3/day. Focus engineering effort elsewhere.

**Phase 2 (100-1,000 negotiations/day):** Still Redpill API, but:
- Monitor cache hit rates (if Redpill exposes `cached_tokens` in usage)
- If latency is a problem, deploy a dedicated SGLang instance on Phala H100 ($74/day)
- Use LFU eviction + HiCache for optimal prefix retention
- Interleave negotiations for GPU utilization

**Phase 3 (1,000+ negotiations/day):**
- Dedicated GPU-TEE cluster (2-4x H100)
- SGLang with RadixAttention + HiCache
- Prefix-aware load balancing across replicas
- Model: Qwen3 30B-A3B (best KV cache efficiency) or GPT-OSS 120B (best reasoning per compute)
- Consider FP8 KV cache quantization (halves KV memory, ~1% quality loss)

### Actual Bottlenecks (Ranked)

For a system doing 10-50 negotiations/day with ~30 turns each:

1. **Engineering time** — building the DAG conversation manager, scope semantics, agent harness. This is 100x more important than inference optimization. **THE bottleneck.**

2. **Model quality** — does the model negotiate well? Can it follow scope rules? This determines whether the product works, not whether it's fast.

3. **Inference latency** — affects UX during demos. Prefix caching helps here. Worth getting right with proper prompt structure, but not worth self-hosting for.

4. **Token cost** — at $0.01-0.03 per negotiation, this is irrelevant at hackathon scale. Would matter at >1,000 negotiations/day.

5. **GPU memory for KV cache** — only matters if self-hosting. Even then, our models + typical context lengths leave ample headroom on H100.

6. **Cache eviction under load** — only matters on shared multi-tenant infrastructure (Redpill). Mitigated by using a dedicated instance or by keeping request frequency high enough to avoid LRU eviction.

7. **Cross-machine KV transfer** — not worth pursuing. Recomputing a 3,000-token prefix (~2 seconds) is cheaper than serializing and transferring 216-960 MB of KV cache over any available interconnect.

### The Honest Summary

Prefix caching is a real optimization that gives meaningful latency improvements for Mnemo's DAG pattern. But at hackathon scale, the economics are dominated by the base API pricing (which is incredibly cheap on Redpill) and the engineering effort to build the product. The KV cache math confirms that our access patterns (rewind, fork, checkpoint) are naturally cache-friendly, and that a properly configured SGLang or vLLM instance will handle them efficiently. The right strategy is: use the API, structure prompts for cache hits, and only self-host if latency becomes a demo-blocking issue.
